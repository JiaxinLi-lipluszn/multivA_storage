C1 · Slack 入口只是管道

*C1 的钩子：别把 SeminarBot 理解成“Slack 机器人里塞了一个大脑”。*
更准确地说：Slack 入口只是管道，负责把人的动作接进来、把结果发出去。真正的判断和事实沉淀被拆到别的模块。
> app.py Slack(Socket Mode)入口 = 管道

SeminarBot 的核心不是一个大文件全包，而是 Slack 管道、判断大脑、git canon 三段分工。

*关键直觉：`app.py` 做的是“接线员”工作。*
它知道 Slack token、Socket Mode、channel、thread、permalink、导入目录、viewer URL、调度参数；这些都是把研讨流程跑起来需要的外壳能力。
> 管道(此文件):收 Slack 事件、发消息、提交 git、调度。

*判断权不在 Slack 入口里。*
成员发来质疑或贡献时，入口只是把消息送去分诊；是否承重、命中哪条 claim、属于什么类型，交给 brain。
> @SeminarBot <你的质疑/贡献> → 大脑分诊(是否承重、打哪条主张、什么类型)→ 写进 canon → 给你打 ✅

*持久事实也不靠 Slack 历史本身。*
Slack 是交互现场；真正可追溯的共识、claim、贡献记录落在 `canon/` 这个 git 仓库里。
> canon.py git 仓库 = 数据库
> canon 的每一次变化都在 `canon/` 里 `git log` 可查

*常见误解：看到 `app.py` 很长，就以为核心逻辑都在入口。*
但这个设计的边界正相反：入口可以复杂，因为 Slack I/O、导入、调度、权限、viewer 都繁琐；“研讨判断”仍被隔离在 brain，“事实源”仍被隔离在 canon。
> 大脑(brain.py):判断,shell 出 codex。
> 存储(canon.py):git 仓库。
